SECTION C – DEEP UNDO, HISTORY, AND DIFFERENTIAL COMPARISON
STATUS: UNDER REVIEW
================================================================================

C.1 Purpose
STATUS: PENDING
Provide safe, precise recovery from mistakes by allowing editors to undo, review,
compare, and selectively restore content changes without rolling back entire pages
or risking unintended regressions.

C.2 Scope
STATUS: PENDING
Applies to:
- Page drafts
- Published page snapshots
- Content blocks
- Section groupings
- Animation configurations
Does NOT apply to:
- Authentication
- Business logic
- External systems

C.3 Requirements

C.3.1 Revision Storage System
STATUS: PENDING

C.3.1.1 Revision Entity Definition
STATUS: PENDING
The system MUST introduce a persistent Revision entity.

C.3.1.1.1 Required Fields
STATUS: PENDING
Each Revision MUST store:
- RevisionId (unique, immutable)
- PageId (foreign key)
- RevisionSequenceNumber (monotonic per PageId)
- RevisionType (DraftSave, PublishSnapshot, SandboxPromotion)
- AuthorUserId
- CreatedUtcTimestamp
- NormalizedBlockTreeSnapshot
- MediaReferenceSnapshot
- AnimationConfigurationSnapshot
- ValidationStateSnapshot

C.3.1.1.2 Indexing Rules
STATUS: PENDING
Indexes MUST exist on:
- PageId + RevisionSequenceNumber
- PageId + CreatedUtcTimestamp

C.3.1.2 Revision Creation Rules
STATUS: PENDING

C.3.1.2.1 Draft Revision on Save
STATUS: PENDING
A draft revision MUST be created when a save operation results in a material change
to content, layout, media, or animations.

C.3.1.2.2 Non-Trigger Conditions
STATUS: PENDING
Revisions MUST NOT be created for:
- Cursor movement
- UI-only state
- Preview rendering
- Validation-only checks

C.3.1.2.3 Failure Handling
STATUS: PENDING
If revision persistence fails:
- The save operation MUST be rejected
- The editor MUST be notified
- No partial revision may remain

C.3.1.3 Publish Snapshot Revisions
STATUS: PENDING
On publish, a PublishSnapshot revision MUST be created and marked immutable.

C.3.2 Diff Computation System
STATUS: PENDING

C.3.2.1 Diff Engine Inputs
STATUS: PENDING
The diff engine MUST accept two Revision snapshots and compute differences across:
- Block additions, removals, and reordering
- Text content changes
- Media reference changes
- Animation parameter changes

C.3.2.2 Diff Output Model
STATUS: PENDING
Diff results MUST be structured, typed, and renderable in UI.

C.3.2.3 Diff Safety Rules
STATUS: PENDING
The diff engine MUST NOT:
- Execute content
- Load external resources
- Modify stored revisions

C.3.3 Restore Operations
STATUS: PENDING

C.3.3.1 Restore Single Block
STATUS: PENDING
Editors MAY restore a single block from a prior revision into the current draft.
After restore:
- A new draft revision MUST be created
- Full validation MUST run

C.3.3.2 Restore Section Group
STATUS: PENDING
Editors MAY restore an entire section grouping following the same rules as block
restore.

C.3.3.3 Restore Failure Handling
STATUS: PENDING
If restore validation fails:
- Restore MUST be rolled back
- User MUST receive actionable errors

C.3.3.4 Cursor Prompt Boundary
STATUS: PENDING
Each restore type maps to a single Cursor prompt:
- One for block restore
- One for section restore
- One for validation enforcement
