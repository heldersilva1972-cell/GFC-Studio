STATUS: UNDER REVIEW

B.1 PURPOSE
STATUS: PENDING
The Multi-Step Publish and Approval Workflow ensures that changes to public-facing
content are deliberate, reviewable, auditable, and reversible. This workflow
prevents accidental publishing, enforces accountability, and supports both simple
and complex editorial organizations.

The workflow is explicit, state-driven, and enforced server-side. No publish action
may rely solely on client-side checks.

B.3 WORKFLOW STATE MODEL
STATUS: PENDING

B.3.1 Defined States
STATUS: PENDING

B.3.1.1 Draft
STATUS: PENDING
Initial editable state where content may be freely modified by authorized editors.

B.3.1.2 PendingApproval
STATUS: PENDING
State indicating content has been submitted for review and is locked from further
editing until approved or rejected.

B.3.1.3 ApprovedReady
STATUS: PENDING
Optional intermediate state indicating content has passed review and is eligible
for immediate or scheduled publish.

B.3.1.4 Published
STATUS: PENDING
State indicating content is live and publicly accessible.

B.3.1.5 Archived
STATUS: PENDING
Terminal state indicating content is no longer active but retained for history.

--------------------------------------------------------------------------------
B.4 STATE TRANSITION RULES
STATUS: PENDING

B.4.1 Draft to PendingApproval
STATUS: PENDING
Transition requires:
• Successful validation (Section A + Section I)
• Submission notes provided by editor
• No active blocking rules

Failure Handling:
• Transition MUST be blocked on validation failure
• Errors MUST be surfaced to editor

B.4.2 PendingApproval to ApprovedReady
STATUS: PENDING
Transition requires:
• Approver role permission
• Optional approval notes
• Immutable snapshot of reviewed content

B.4.3 PendingApproval to Draft (Rejection)
STATUS: PENDING
Rejection requires:
• Mandatory rejection reason
• Automatic unlock for editor
• Preservation of rejection audit trail

B.4.4 ApprovedReady to Published
STATUS: PENDING
Transition requires:
• Final validation pass
• Renderer compatibility confirmation
• Optional scheduling constraints satisfied

B.5 APPROVAL QUEUE AND REVIEW PROCESS
STATUS: PENDING

B.5.1 Review Queue
STATUS: PENDING
The system MUST present approvers with a centralized queue listing:
• Page title
• Current version
• Submitter
• Submission timestamp
• Validation summary

B.5.2 Review Actions
STATUS: PENDING
Approvers may:
• Approve
• Reject
• Request changes (non-terminal)

All actions MUST generate audit events.

--------------------------------------------------------------------------------
B.6 PUBLISH SCHEDULING
STATUS: PENDING

B.6.1 Scheduling Capability
STATUS: PENDING
Approved content MAY be scheduled for future publication.

B.6.2 Scheduling Rules
STATUS: PENDING
• Only ApprovedReady content may be scheduled
• Scheduling may be cancelled prior to execution
• Scheduling MUST respect maintenance windows

B.6.3 Failure Handling
STATUS: PENDING
If scheduled publish fails:
• Content remains in ApprovedReady
• Failure is logged and surfaced
