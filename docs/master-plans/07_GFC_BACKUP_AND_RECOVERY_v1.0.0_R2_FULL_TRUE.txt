SECTION 07 – BACKUP & RECOVERY
ECOSYSTEM VERSION: v1.0.0

================================================================================
PURPOSE OF THIS DOCUMENT
================================================================================
This document defines the authoritative Backup & Recovery strategy for the
entire GFC Ecosystem. It is designed to be uploaded into any future chat and
allow recovery planning, validation, and implementation with zero ambiguity.

This is a FULL STANDALONE document (R1 baseline).

================================================================================
SECTION A – SYSTEMS IN SCOPE
================================================================================
The Backup & Recovery strategy applies to ALL of the following systems:

1. GFC Web App
2. GFC Web Studio
3. GFC New Website (public-facing)
4. GFC Project Tracker
5. Animation Playground (as integrated into Studio)
6. Databases (all environments)
7. Static assets (images, documents, media)
8. Configuration and secrets (securely handled)

================================================================================
SECTION B – CORE PRINCIPLES (EXPLICIT)
================================================================================

B1. Recoverability over Convenience
- Backups exist to enable recovery, not just storage.
- Restore procedures are as important as backup procedures.

B2. Non-Destructive Restoration
- Restores must not silently overwrite newer data.
- Explicit confirmation required for destructive restores.

B3. Least Privilege
- Only authorized roles may trigger backup or restore operations.

B4. Auditability
- All backup and restore actions are logged.

================================================================================
SECTION C – BACKUP TYPES
================================================================================

C1. Application Code Backups
- Source repositories (Git) are primary source of truth.
- Periodic snapshot backups of deployed code are still required.

C2. Database Backups
- Full database backups
- Incremental / differential backups (if supported)
- Transaction log backups (where applicable)

C3. Content & Asset Backups
- Studio-authored content
- Uploaded images and documents
- Generated animation definitions

C4. Configuration Backups
- Environment configuration
- Feature flags
- Deployment settings

================================================================================
SECTION D – BACKUP FREQUENCY & RETENTION (BASELINE)
================================================================================

D1. Databases
- Daily full backup
- Hourly incremental (or equivalent)
- Retention:
  - Daily: 30 days
  - Monthly: 12 months
  - Yearly: 7 years (if required)

D2. Assets & Content
- Daily snapshot
- Retention: 90 days

D3. Configuration
- Backup on every change
- Retention: full history

================================================================================
SECTION E – STORAGE STRATEGY
================================================================================

E1. Primary Backup Storage
- Off-server storage (not same host as production)
- Encrypted at rest

E2. Secondary / Redundant Storage
- Optional second location
- Protects against single-location failure

E3. Access Controls
- Read access restricted
- Restore access restricted to SuperAdmin

================================================================================
SECTION F – RESTORE PROCEDURES
================================================================================

F1. Database Restore
- Restore to new instance first
- Validate integrity
- Promote to production only after confirmation

F2. Application Restore
- Restore code snapshot
- Validate environment compatibility

F3. Content Restore
- Restore Studio content independently of DB where possible
- Validate render output before publish

F4. Partial Restore
- Restore single module or dataset when possible
- Avoid full system restore if not required

================================================================================
SECTION G – DISASTER RECOVERY SCENARIOS
================================================================================

G1. Data Corruption
- Identify last-known-good backup
- Restore to staging
- Validate before production

G2. Deployment Failure
- Roll back to last stable deployment
- Restore config if needed

G3. Security Incident
- Lock system
- Restore from trusted backup
- Rotate credentials

================================================================================
SECTION H – TESTING & VALIDATION
================================================================================

H1. Backup Validation
- Periodic verification that backups exist and are readable

H2. Restore Drills
- Scheduled restore tests (non-production)
- Document results

================================================================================
SECTION I – ROLES & RESPONSIBILITIES
================================================================================

- SuperAdmin:
  - Configure backups
  - Approve restores
- Admin:
  - View backup status
- ReadOnly:
  - No access

================================================================================
SECTION J – NON-GOALS
================================================================================

- Backups are not a replacement for source control.
- Backup systems do not perform active monitoring.
- Automatic destructive restores are forbidden.

SECTION K – ENVIRONMENT SEPARATION
---------------------------------
Defined environments:
- Development
- Staging / Test
- Production

Rules:
- Production backups are immutable and highest priority.
- Production restores require SuperAdmin approval and confirmation.
- Production → Staging restores are allowed.
- Staging/Dev → Production restores are forbidden.

SECTION L – OWNERSHIP & RESPONSIBILITY MATRIX
---------------------------------------------
Artifacts and ownership:

- Databases: Owner = SuperAdmin
- Studio Content: Owner = ContentAdmin
- Website Assets: Owner = ContentAdmin
- Configuration: Owner = SuperAdmin

Restore authority:
- Only SuperAdmin may execute restores.
- Admin may view status only.

SECTION M – FAILURE MODES & HANDLING
-----------------------------------
Failure scenarios:
- Backup job fails → log + alert
- Storage unavailable → retry + escalate
- Retention cleanup fails → alert
- Restore validation fails → abort restore

All failures must generate audit entries.

SECTION N – ENCRYPTION & SECRETS
--------------------------------
- All backups encrypted at rest.
- Encryption keys owned by SuperAdmin.
- Secrets are not stored in plaintext backups.
- Key rotation required periodically.

SECTION O – VERSION & COMPATIBILITY RULES
-----------------------------------------
- App version must match DB schema version.
- Studio schema must match Website renderer.
- Migrations required before restore if mismatched.

SECTION P – LAST KNOWN GOOD STATE
--------------------------------
Defined as:
- Successful publish
- Verified backup
- No active errors

Marker stored with backup metadata.

SECTION Q – SIMULATION / DRY-RUN RESTORES
-----------------------------------------
- Dry-run restore validates integrity only.
- No data overwritten.
- Used for testing and audits.

SECTION R – WEB APP UI (DESIGN INTENT)
-------------------------------------
Web App should expose:
- Backup status dashboard
- Last backup timestamps
- Manual trigger buttons (with confirmation)
- Restore history
- Warnings for destructive actions

SECTION S – CURSOR PROMPT PATHS
-------------------------------
Required prompt groups:
1. Backup service setup
2. Storage configuration
3. Restore tooling
4. UI dashboard
5. Audit logging

SECTION T – EXPLICIT EXCLUSIONS
-------------------------------
Not backed up:
- Source repositories (Git managed)
- Runtime caches
- Temporary build artifacts
