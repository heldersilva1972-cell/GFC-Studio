PURPOSE OF THIS DOCUMENT
================================================================================
This document defines the GFC Portal as a first-class, standalone system within
the GFC Ecosystem. It is written to be uploaded into any future chat and allow
work to continue with zero ambiguity, zero guessing, and zero reliance on memory.

The Portal is the authenticated, role-aware entry layer that routes users to the
correct tools (Web App, Studio, dashboards) while enforcing security, visibility,
and safe operation.

• FULL

================================================================================
SECTION A – PORTAL SCOPE & INTENT (NON-NEGOTIABLE)
================================================================================
The GFC Portal exists to provide a unified, secure entry point for all internal
users of the GFC Ecosystem.

The Portal:
• Authenticates users
• Enforces session and security state
• Routes users to authorized systems
• Provides system visibility and awareness
• Aggregates status and alerts (read-only)

The Portal does NOT:
• Contain business logic
• Modify data directly
• Replace the Web App
• Replace Studio
• Render public website content
• Execute destructive actions

================================================================================
SECTION B – USER TYPES & ROLE AWARENESS
================================================================================
The Portal supports the following roles (defined by Web App):

• SuperAdmin
• Admin
• ContentEditor
• Finance
• ReadOnly

Rules:
• Roles are authoritative from the Web App
• Portal may never infer or override roles
• UI adapts dynamically based on role

================================================================================
SECTION C – ROLE-BASED LANDING PAGES
================================================================================
After authentication, the Portal routes users to a role-appropriate landing view.

SuperAdmin:
• System health overview
• Backup status (read-only)
• Security alerts
• Quick access to Web App & Studio

Admin:
• Admin dashboards
• Content and site status
• Alerts summary

ContentEditor:
• Studio launch
• Content publish status
• Draft warnings

Finance:
• Financial dashboards
• Reports access

ReadOnly:
• View-only dashboards
• No action controls

================================================================================
SECTION D – UNIFIED NAVIGATION SHELL
================================================================================
The Portal provides a consistent navigation shell across all tools.

Navigation responsibilities:
• Persistent header/sidebar
• Consistent user identity display
• Tool switching without re-authentication
• Clear separation of tools

Rules:
• Navigation never bypasses permissions
• Disabled tools show reason, not errors

================================================================================
SECTION E – SYSTEM STATUS AGGREGATION (READ-ONLY)
================================================================================
The Portal aggregates non-actionable system status:

• Backup last run time
• Website publish state
• Studio availability

Rules:
• No execution actions allowed
• Status sourced from authoritative systems

================================================================================
SECTION F – ALERT & NOTIFICATION INBOX
================================================================================
The Portal exposes a centralized alert inbox.

Alert types:
• Security warnings
• Backup failures
• System degradations
• Policy violations

Rules:
• Critical alerts cannot be dismissed
• Alerts link to authoritative system views
• Portal does not resolve alerts

================================================================================
SECTION G – CROSS-SYSTEM CONTEXT PASSING
================================================================================
When launching tools, the Portal passes context:

• User identity
• Role
• Launch intent (page, project, task)

Rules:
• Context is advisory, not authoritative
• Target system validates permissions

================================================================================
SECTION H – SESSION & SECURITY GATEWAY ROLE
================================================================================
The Portal is the primary session holder.

Responsibilities:
• MFA enforcement state
• Session expiration
• Global logout propagation
• Lockout enforcement

Rules:
• Security & Hardening overrides Portal behavior
• Portal cannot weaken security controls

================================================================================
The Portal is aware of feature flags to:

• Hide unavailable tools
• Display “coming soon” safely
• Prevent broken navigation

Rules:
• Flags managed only by Web App
• Portal is read-only consumer

Rules:

SECTION K – AUTHORITY RESOLUTION RULES
================================================================================
The Portal never decides truth.

Authority map:
• Identity & roles → Web App
• Content → Studio
• Security → Section 08
• Backups → Section 07

In conflicts, the Portal defers and surfaces warnings.
